# Product

## Register

product

## Users

Prism serves developer/operators who connect Slack to developer-owned local tools. They are configuring trusted access for CLIs, MCP servers, coding agents, and internal automation, often while thinking about credential safety, workspace scope, and least-privilege behavior.

## Product Purpose

Prism is a hosted Slack-compatible bridge that keeps Slack credentials in server custody while local tools receive only opaque Prism developer tokens. The website helps users link Slack, create and manage Token profiles, understand what each profile can do, and inspect metadata-only activity without exposing Slack message contents, file contents, search text, or credentials.

Success means a developer can quickly answer three questions: is Slack linked and healthy, which local tools have access, and what has Prism forwarded or blocked.

## Brand Personality

Professional but playful, leaning into the Prism metaphor: precise, secure, lucid, and refractive without feeling gimmicky. The interface should feel trustworthy enough for credential custody and polished enough for daily developer workflows.

## Anti-references

Avoid generic flashy SaaS landing pages, ad hoc inline styling, inconsistent buttons, decorative cyberpunk dark mode, and dashboards that hide critical safety information behind vague marketing copy. Avoid making Token profile creation feel like a raw debug form.

## Design Principles

1. Make trust visible: show custody boundaries, policy outcomes, and audit behavior in plain language.
2. Optimize for safe action: Token profile creation, rotation, and revocation should feel deliberate, fast, and reversible where appropriate.
3. Use the Prism metaphor with restraint: light, facets, and spectrum details can guide hierarchy, but the product must stay task-first.
4. Keep Slack content out of sight: reinforce that Prism stores metadata only and never surface message or token material accidentally.
5. Prefer earned familiarity: use proven product UI patterns, consistent controls, and clear status language over invented affordances.

## Accessibility & Inclusion

Target WCAG AA. The interface should support keyboard-first use, visible focus states, reduced-motion preferences, color-blind-safe status indicators, sufficient contrast, and no hover-only affordances.
