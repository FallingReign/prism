# Prism security review notes

These notes describe Prism v1's current security posture for developers and reviewers. Current code and runtime behavior are the source of truth if wording drifts.

## Credential custody

The **Prism hosted service** owns Slack OAuth callbacks and stores **Slack credentials** only server-side. A **Local tool** receives a **Prism developer token**, not Slack app credentials, bot tokens, user tokens, refresh tokens, or app-level tokens.

Slack credential envelopes are encrypted before persistence. Local development uses encrypted envelopes in Postgres and `.env.local` secret material generated by migration helpers. Non-local environments should use approved secret storage and managed KMS-equivalent key custody for encryption keys, peppers, Slack client secrets, signing secrets, and database credentials.

Do not commit, print, or paste Slack credentials, Prism developer tokens, token hashes, peppers, or Authorization headers into docs, issues, screenshots, prompts, or logs.

## Prism developer token risk

A Prism developer token is an opaque bearer secret mapped server-side to exactly one **Token profile**. Treat it like a password for the permitted Capability map.

Mitigations:

- issue one Token profile per Local tool or purpose
- choose the narrowest useful Capability map
- set expiry where practical
- rotate tokens regularly
- revoke tokens immediately after suspected token theft
- narrow policy when a Local tool no longer needs a capability
- avoid copying tokens into shell history, terminal recordings, or LLM prompts

Policy broadening requires confirmation and replacement token rotation. Rotation may be immediate or use a bounded overlap for migration.

## Metadata-only audit

**Metadata-only audit** records user/profile/workspace/method/category/status/error/request ID and selected object metadata. It must not store Slack message bodies, DMs, search results, file contents, Block Kit payloads, canvases, lists, Prism developer tokens, token hashes, peppers, Slack credentials, refresh tokens, access tokens, or client secrets.

The Prism website can show audit review data, but audit entries are not a Slack archive and do not provide payload logging.

## Local execution risk

Local tools, MCP servers, CLIs, coding agents, and custom applications run outside the Prism hosted service. Prism can enforce Token profile policy and keep Slack credentials server-side, but it cannot make local execution safe by itself.

Run Local tools in the least-privileged environment practical. Review what a coding agent can read, write, execute, and exfiltrate. Treat terminal output, MCP tool results, and generated patches as sensitive when they may include Slack-derived content.

## Prompt-injection risk

Slack content returned through Slack-compatible endpoints is untrusted input. Messages, names, channel topics, search results, and file metadata can contain instructions aimed at coding agents or humans.

Prism does not provide content moderation in v1. Local tools and agents must not treat Slack content as trusted instructions. Prefer explicit user confirmation before using Slack-derived content to modify code, run commands, send messages, or broaden Token profile policy.

## Rate limits and operational controls

Prism-side rate limits are per Token profile and Slack method. A Prism-side 429 returns `error: "rate_limited"`, `Retry-After`, and `X-Prism-Upstream-Called:false`.

Upstream Slack rate limits remain Slack-owned. An upstream Slack 429 preserves Slack's response body and retry headers where practical, and `X-Prism-Upstream-Called:true` shows Prism did call upstream.

Operational controls:

- monitor health with `/v1/prism/health`
- run migrations before local or deployment verification
- use `/v1/prism/status` and `/v1/prism/capabilities` before exposing Local tool actions
- review Metadata-only audit for suspicious activity
- rotate or revoke Token profiles after abnormal usage
- keep Slack admin approval records outside source control

## Explicit v1 deferrals

Prism v1 excludes inbound events, Socket Mode fanout, slash commands, interactivity, app mentions, file transfer, canvases, lists, payload logging, content moderation, Supabase platform services, and Slack administration.

Prism is not an unmanaged Slack proxy. The **Method registry** and **Capability map** decide what a **Slack-compatible endpoint** may do before forwarding. Slack administration still owns workspace/org policy and the maximum approved app capability.
